## Overview

Four problems. Opgave 1 covers data-structure mechanics: Huffman tree, max-heap insert, hashing with linear probing. Opgave 2 is greedy graph algorithms: an MST cut/cycle proof plus tracing an MST and a shortest-path tree. Opgave 3 is augmented red-black trees (store subtree max y, support a range query in O(log n)). Opgave 4 is dynamic programming on a sorted sequence (minimise a sum-of-squared-gap-error objective).

Danish terms kept in parentheses where useful: letteste udspændende træ = minimum spanning tree (MST), kreds = cycle, korteste-vej-træ = shortest-path tree, hob = heap, ujævnhed = unevenness/cost.

## Opgave 1 (20%)

### 1a — Huffman tree (8%)

**Problem.** Alphabet `a,b,c,d,e,f,g` with frequencies a=9, b=7, c=24, d=10, e=55, f=15, g=25. Draw a Huffman tree (Huffman-træ) representing the Huffman codes.

**Type.** Build a Huffman tree by repeated greedy merge of the two smallest weights.

**Method (any instance).**
1. Put every symbol in a min-priority queue keyed by frequency.
2. Repeat until one node remains: extract the two smallest nodes `x,y`, make a new internal node with weight `x+y` and children `x,y`, insert it back.
3. The last node is the root. Label left edges 0, right edges 1; a symbol's code is the root-to-leaf string. Swap in your own frequency table at step 1.

**Worked solution.** Merges (always the two current smallest):
1. b(7) + a(9) = 16
2. d(10) + f(15) = 25
3. (ba)(16) + c(24) = 40
4. g(25) + (df)(25) = 50
5. ((ba)c)(40) + (g(df))(50) = 90
6. e(55) + (((ba)c)(g(df)))(90) = 145 = root

Tree (weights in parentheses):

```
                 (145)
                /     \
              e(55)   (90)
                     /    \
                 (40)      (50)
                /   \      /   \
            (16)   c(24) g(25) (25)
            /  \              /   \
          b(7) a(9)         d(10) f(15)
```

Codes (left=0, right=1), read off the paths (checked: Python):
- e=`0` (length 1)
- c=`101` (length 3): (90)→(40)(left)/(50)(right); (40)→(16)(left)/c(right).
- b=`1000`, a=`1001` (length 4): (16)→b(left)/a(right).
- g=`110` (length 3): (50)→g(left)/(25)(right).
- d=`1110`, f=`1111` (length 4): (25)→d(left)/f(right).

So code lengths are e=1, c=3, g=3, b=a=d=f=4.

Total bits = 9·4 + 7·4 + 24·3 + 10·4 + 55·1 + 15·4 + 25·3 = 36+28+72+40+55+60+75 = 366 (checked: Python). The 0/1 assignment per edge is not unique; any consistent labelling that keeps the merge structure is correct. The point asked for the tree.

### 1b — Insert into max-heap (7%)

**Problem.** Insert key 9 into the binary max-heap (max-hob) stored as array `[10, 8, 6, 3, 7, 4, 5, 1, 2]`. Show the array before each pass of the while-loop of MAX-HEAP-INSERT (CLRS p.140 / HEAP-INCREASE-KEY sift-up). May draw as a tree.

**Type.** Trace heap insert = append, then sift-up while parent < element.

**Method.** 1. Append new key at the next array slot (a new leaf). 2. Let `i` be its 1-based index. 3. While `i>1` and `A[parent(i)] < A[i]`: swap with parent, set `i = parent(i)`. `parent(i)=⌊i/2⌋`. Show the array at the top of each while test.

**Worked solution.** Append 9 at index 10: `[10, 8, 6, 3, 7, 4, 5, 1, 2, 9]`.

- Before iteration 1 (i=10): `[10, 8, 6, 3, 7, 4, 5, 1, 2, 9]`. parent(10)=5, A[5]=7 < 9 → swap.
- Before iteration 2 (i=5): `[10, 8, 6, 3, 9, 4, 5, 1, 2, 7]`. parent(5)=2, A[2]=8 < 9 → swap.
- Before iteration 3 (i=2): `[10, 9, 6, 3, 8, 4, 5, 1, 2, 7]`. parent(2)=1, A[1]=10 > 9 → stop.

Final heap: `[10, 9, 6, 3, 8, 4, 5, 1, 2, 7]` (checked: Python). As a tree: root 10; children 9, 6; 9's children 3, 8; 6's children 4, 5; 3's children 1, 2; 8's child 7.

### 1c — Hashing with linear probing (5%)

**Problem.** Table of 10 slots, occupied: slot1=11, slot2=31, slot4=24, slot5=15, slot8=48. Hash `h(k)=k mod 10`. Insert key 4 — which slot?

**Type.** Linear probing insert: start at `h(k)`, step +1 mod m until empty.

**Method.** Compute `i=h(k)`. While slot `i` occupied, `i=(i+1) mod m`. Insert at first empty `i`.

**Worked solution.** `h(4)=4`. Slot 4 = 24 (occupied) → slot 5 = 15 (occupied) → slot 6 = empty. **Key 4 lands in slot 6.**

### 2a — Heaviest-edge-in-a-cycle is not in any MST (9%)

**Problem.** `G` weighted, `e` an edge. Suppose there is a cycle (kreds) `C` with `e ∈ C` and `w(e) > w(e')` for every other edge `e'` of `C`. Prove `e` cannot be in any MST.

**Type.** Cycle property of MSTs, proof by exchange argument.

**Method (cut/exchange proof).** 1. Assume for contradiction an MST `T` contains `e`. 2. Removing `e` splits `T` into two components `S` and `V−S`. 3. Find a cheaper edge of the cycle crossing the same cut. 4. Swap it in to get a spanning tree of smaller weight — contradiction.

**Worked solution.** Assume some MST `T` contains `e=(u,v)`. Delete `e` from `T`: `T` splits into two trees, with vertex sets `S` (containing `u`) and `V−S` (containing `v`). Now look at the cycle `C`. Walk around `C` starting from `u` and ending at `v` along the path that does not use `e`. This path starts in `S` (at `u`) and ends in `V−S` (at `v`), so at some point it crosses the cut, i.e. `C` contains an edge `e' ≠ e` with one endpoint in `S` and the other in `V−S`. By assumption `w(e') < w(e)` (since `e` is strictly the heaviest edge of `C`). Then `T' = T − e + e'` reconnects the two components (both endpoints of `e'` are on opposite sides), so `T'` is again a spanning tree, and `w(T') = w(T) − w(e) + w(e') < w(T)`. That contradicts `T` being minimum. Hence no MST contains `e`. ∎

### 2b — Draw an MST and argue (8%)

**Type.** Trace Kruskal (or Prim); argue via the cut property / uniqueness.

**Method.** Run Kruskal: sort edges ascending, add an edge if it joins two different components (no cycle), stop at `n−1` edges. Argue correctness: each added edge is the lightest edge crossing the cut between its two components, so by the cut property it is safe.

**Worked solution.** Kruskal order (n−1 = 12 edges added):

| step | edge | w | action |
|---|---|---|---|
| 1 | s–A | 1 | add |
| 2 | Q–R | 2 | add |
| 3 | D–E | 3 | add |
| 4 | G–I | 4 | add |
| 5 | B–G | 5 | add |
| 6 | H–I | 6 | add |
| 7 | C–D | 7 | add |
| 8 | R–C | 8 | add |
| – | G–H | 8 | skip (cycle G–I–H) |
| 9 | P–Q | 9 | add |
| – | P–R | 10 | skip (cycle) |
| 10 | D–F | 11 | add |

| – | A–B | 13 | skip (cycle s–A–B) |
| – | E–F | 14 | skip (cycle) |
| – | R–D | 15 | skip (cycle in right part) |
| 12 | F–I | 15 | add (bridges left and right halves) |
| – | C–G | 17 | skip |
| – | B–C | 30 | skip |

**MST edges:** `s-A(1), Q-R(2), D-E(3), G-I(4), B-G(5), H-I(6), C-D(7), R-C(8), P-Q(9), D-F(11), s-B(12), F-I(15)`. **Total weight = 83** (checked: Python).

Uniqueness note: weights repeat only at 8 (R–C and G–H) and 15 (R–D and F–I). In each pair Kruskal uses one and the other forms a cycle, and there is no edge of weight < 15 bridging the two halves of the graph (the left set `{s,A,B,G,H,I}` and the right set `{P,Q,R,C,D,E,F}` are only joined by C–G=17, B–C=30, or F–I=15), so F–I=15 is forced. The MST is unique. Each chosen edge is the minimum-weight edge crossing the cut separating the component built so far, so by the cut property the tree is a minimum spanning tree.

### 2c — Shortest-path tree from s (8%)

**Type.** Trace Dijkstra; the tree is given by the `prev`/predecessor pointers.

**Method.** Run Dijkstra from `s`: init `d[s]=0`, others ∞; repeatedly extract the unsettled vertex of smallest `d`, relax its edges (`if d[u]+w < d[v]: d[v]=d[u]+w, prev[v]=u`). The shortest-path tree is the set of edges `(prev[v], v)`.

**Worked solution.** Distances from `s` and the predecessor (tree parent):

| node | dist | via (parent) |
|---|---|---|
| s | 0 | – |
| A | 1 | s |
| B | 12 | s |
| G | 17 | B |
| I | 21 | G |
| H | 25 | G |
| C | 34 | G |
| F | 36 | I |
| D | 41 | C |
| Q | 44 | R |
| E | 44 | D |
| R | 42 | C |
| P | 52 | R |

So the shortest-path-tree edges (rod/root = s): `s-A(1), s-B(12), B-G(5), G-I(4), G-H(8), G-C(17), I-F(15), C-D(7), C-R(8), D-E(3), R-Q(2), R-P(10)`.

Sample paths: `s→B→G→C` = 12+5+17 = 34 (cheaper than `s→B→C` = 12+30 = 42). `s→B→G→I→F` = 12+5+4+15 = 36 (cheaper than via D: 41+11=52). `s→…→C→R` = 34+8 = 42, then `R→P` = 52 (note `P→R→C…` beats `P→Q` path because Q itself is reached via R at 44). Argue correctness: Dijkstra settles vertices in nondecreasing distance order and never revisits a settled vertex, valid because all edge weights are nonnegative; when a vertex is extracted its `d` value is final, so the predecessor edges form a correct shortest-path tree.

## Opgave 3 (30%)

Points in the plane, positive integer coordinates, all x-coordinates distinct. `MinAbove(t)`: among all points with `y ≥ t`, return the one with smallest `x`; if none, report "none". Stored in a red-black tree keyed by x; each node also stores `ymax` = the maximum y-coordinate in its subtree.

### 3a — Evaluate MinAbove(10) (5%)

**Problem.** Points (4,9),(5,17),(19,6),(23,10),(25,15),(40,7). Compute `MinAbove(10)`.

**Type.** Direct evaluation of the query.

**Worked solution.** Points with `y ≥ 10`: (5,17), (23,10), (25,15). Smallest x among these = 5. **MinAbove(10) = (5,17).** (Matches the circled point in the figure.)

### 3b — Maintain the augmented tree under insert/delete (10%)

**Type.** Augmenting a red-black tree with a subtree aggregate (CLRS Ch.14 augmentation theorem).

**Method (general recipe for any subtree aggregate `f`).**
1. The extra field is determined by the node's own value and its two children's fields: here `x.ymax = max(x.y, x.left.ymax, x.right.ymax)` (treat a missing child as `−∞`). This is the key property — the field of a node depends only on the node and its children.
2. **Insert / delete:** do the normal BST insert/delete. On the way down (or up) recompute `ymax` for every node on the path, since their subtrees changed. Cost O(log n) along one path.
3. **Rotations:** a rotation only changes the two nodes directly involved and their subtree contents are reshuffled only between those two. Recompute `ymax` for the lower node first, then the higher node, from their (new) children. O(1) per rotation, O(log n) rotations total.
4. By the augmentation theorem (CLRS Thm 14.1), because `ymax` depends only on a node and its children, it can be maintained without changing the asymptotic RB-tree bounds: insert and delete stay O(log n).

**Worked solution / description.** Store at each node `v` the value `v.ymax = max(v.y, left.ymax, right.ymax)`. During the descending search of INSERT, after attaching the new leaf, walk back up to the root recomputing `ymax` at each ancestor (the new point may raise some `ymax` values). During the RB fix-up, each rotation (LEFT-ROTATE/RIGHT-ROTATE) recomputes `ymax` of the node that moved down, then the node that moved up, using their children — O(1) each. DELETE is symmetric: after splicing out the node (and possibly moving the successor), recompute `ymax` up the affected path and after every fix-up rotation. All work is O(log n), so the augmented tree keeps RB insert/delete at O(log n).

### 3c — Insert (30,11) — illustrate (7%)

**Type.** Apply the 3b maintenance to a concrete insert.

**Worked solution.** Tree before (x,y over ymax), root (19,6)/17:
```
                 (19,6) ymax=17
                /              \
       (4,9) ymax=17        (25,15) ymax=15
              \              /        \
        (5,17) ymax=17  (23,10) y=10  (40,7) y=7
```
Insert key x=30. BST search: 30 > 19 → right to (25,15); 30 > 25 → right to (40,7); 30 < 40 → left child of (40,7), which is empty → put (30,11) there as a red leaf. New leaf `ymax = 11`.

Now update `ymax` up the path:
- (40,7): max(7, child (30,11).ymax=11) = **11** (was 7).
- (25,15): max(15, left (23,10)=10, right (40,7)=11) = **15** (unchanged).
- (19,6): max(6, left 17, right 15) = **17** (unchanged).

RB color check: parent (40,7) is black, so no red-red violation; no rotation needed. Resulting subtree:
```
        (25,15) ymax=15
        /            \
 (23,10) y=10     (40,7) ymax=11
                   /
            (30,11) y=11 (red)
```
(If the inserted node had created a red-red conflict, the relevant rotation would also recompute `ymax` for the two rotated nodes — none needed here.)

### 3d — MinAbove(t) in O(log n) (8%)

**Type.** Tree search guided by the augmented `ymax` field.

**Method / algorithm.** Use `ymax` to prune. We want the smallest-x point with `y ≥ t`. Recurse from the root, always preferring the left (smaller x) side when it can contain a qualifying point:

```
MinAbove(node, t):
  if node == NIL or node.ymax < t:      # whole subtree has all y < t
      return NONE
  if node.left != NIL and node.left.ymax >= t:
      return MinAbove(node.left, t)     # answer is in left subtree (smaller x)
  if node.y >= t:
      return node                       # node itself beats everything to its right
  return MinAbove(node.right, t)        # otherwise must be to the right
```

**Why O(log n).** The first guard rejects an entire subtree in O(1) using `ymax`. At each node we recurse into exactly one child, so we follow a single root-to-leaf path — O(log n) total because the RB tree has height O(log n).

**Why correct.** Points are ordered by x left-to-right (BST on x). If the left subtree contains any point with `y ≥ t` (detected by `left.ymax ≥ t`), the answer must be there, because every such point has smaller x than the node and everything in the right subtree. If the left subtree has none but the node itself qualifies, the node is the smallest-x qualifying point (everything left of it fails). Otherwise the answer, if any, lies in the right subtree. The top `ymax < t` test handles "no qualifying point exists" → return NONE.

## Opgave 4 (25%)

Sorted sequence of positive integers `X = ⟨x₁,…,xₙ⟩` and integer `W`. Find a subsequence `Y = ⟨y₁,…,yₘ⟩` with `y₁ = x₁`, `yₘ = xₙ`, minimising the unevenness (ujævnhed) `U = Σ_{i=1}^{m−1} (y_{i+1} − y_i − W)²`. `U(k)` = minimum unevenness of a subsequence of `⟨x₁,…,x_k⟩` (so `y₁=x₁`, ends at `x_k`). `U(1)=0`.

### 4a — Fill the table (10%)

**Problem.** `X = ⟨2,6,8,10,14,15,20,21⟩`, `W=7`. Fill `U(k)` for k=1..8.

**Type.** Fill a 1-D DP table from the recurrence below.

**Recurrence.** A subsequence ending at `x_k` picks some previous element `x_j` (j<k) as the predecessor:
`U(k) = min_{1 ≤ j < k} [ U(j) + (x_k − x_j − W)² ]`, with `U(1)=0`.

**Worked solution.** With `W=7`:

| k | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 |
|---|---|---|---|---|---|---|---|---|
| X[k] | 2 | 6 | 8 | 10 | 14 | 15 | 20 | 21 |
| **U(k)** | **0** | **9** | **1** | **1** | **2** | **1** | **3** | **2** |

Spot checks: `U(2)=(6−2−7)²=9`. `U(3)=min(U(1)+(8−2−7)²=1, U(2)+(8−6−7)²=9+25=34)=1`. `U(8)=2` realised by predecessor `x₅=14`: `U(5)+(21−14−7)² = 2+0 = 2`.

### 4b — A minimum-unevenness subsequence (5%)

**Type.** Backtrack the DP choices from k=n.

**Worked solution.** Follow the argmin predecessors from k=8: 8←5←3←1, i.e. x-values 21,14,8,2. **`Y = ⟨2, 8, 14, 21⟩` with `U = (8−2−7)² + (14−8−7)² + (21−14−7)² = 1 + 1 + 0 = 2`.** (Ties may give another optimum; any subsequence with U=2 is correct.)

### 4c — DP algorithm + running time/space (10%)

**Type.** Give a DP algorithm for `U(n)` and state complexity.

**Algorithm.**
```
DP-Unevenness(X[1..n], W):
  U[1] = 0
  for k = 2 to n:
      U[k] = +inf
      for j = 1 to k-1:
          c = U[j] + (X[k] - X[j] - W)^2
          if c < U[k]: U[k] = c
  return U[n]
```
Optimal substructure: an optimal subsequence ending at `x_k` consists of an optimal subsequence ending at its second-to-last element `x_j` plus the term `(x_k−x_j−W)²`; if the prefix were not optimal we could swap in a better one and lower `U`. So the recurrence is exact.

**Running time:** the double loop does Σ_{k=2}^{n}(k−1) = O(n²) constant-work iterations → **O(n²) time**.
**Space:** one array `U[1..n]` → **O(n) space** (add a `pred[1..n]` array, still O(n), if you also want to recover the subsequence as in 4b).
