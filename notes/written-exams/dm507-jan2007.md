## Opgave 1 (20%) — Binary search trees

### a (5%): Is this a binary search tree?

**Problem.** The tree: root 4; left child 2 (its left child 1); right child 20 with left child 8 and right child 25; node 8 has left child 7 and right child 14; node 14 has left child 11. Is it a BST (binært søgetræ)?

**Type.** Verify the BST property.

**Method (any instance).**
1. The BST property must hold *globally*, not just parent vs child: every key in a node's left subtree is `<` the node's key, every key in the right subtree is `>` it.
2. Fastest check: do an in-order traversal (left, node, right). The result is sorted iff the tree is a BST.
3. If it is not sorted, the first out-of-order pair pinpoints the violation.

**Worked solution.** In-order traversal: `1, 2, 4, 7, 8, 11, 14, 20, 25` (checked: Python). This is strictly increasing, so **yes, it is a binary search tree**. (Sanity points: 8 and 14 sit in the left subtree of 20 and both are `< 20` and `> 4`; 11 `< 14` and `> 8`. All fine.)

### b (8%): Delete the node with key 5 (CLRS p. 261)

**Problem.** BST: root 5; left 2 (children 1 and 4, where 4 has left child 3); right 11 (left child 6, right child 12); node 6 has right child 8; node 8 has children 7 and 10. Draw the tree after deleting key 5 using the textbook `TREE-DELETE`.

**Type.** Trace CLRS `TREE-DELETE` (node with two children).

**Method (any instance).** For deleting node `z`:
1. If `z` has no left child → transplant `z` with `z.right`.
2. If `z` has no right child → transplant `z` with `z.left`.
3. Else (two children): let `y = TREE-MINIMUM(z.right)` (the successor = leftmost node of the right subtree).
   - If `y` is not `z`'s direct child, transplant `y` with `y.right`, then set `y.right = z.right` and fix parent.
   - Transplant `z` with `y`, set `y.left = z.left`, fix parent.
4. `TRANSPLANT(u,v)` replaces the subtree rooted at `u` by the subtree rooted at `v` in `u`'s parent link.

**Worked solution.** `z = 5` has two children. Right subtree is rooted at 11; its minimum is `6` (11's left child, and 6 has no left child). So successor `y = 6`. Since 6 is not the direct child of 5, but 6 *does* have a right child 8: transplant 6 with 8 (so 8 becomes 11's left child), then `6.right = 11`. Finally 6 replaces 5 as the root and `6.left = 2`.

Resulting tree:
```
            6
          /   \
        2       11
       / \     /  \
      1   4   8    12
         /   / \
        3   7   10
```
Root `6`; left = `2` (children `1`, `4`; node `4` has left child `3`); right = `11` (left child `8` with children `7`,`10`; right child `12`).

### c (7%): Insert a node with key 16 (CLRS p. 262)

**Problem.** BST: root 10; left 5 (left child 3); right 21 with left child 15 and right child 30; node 15 has children 11 and 17. Draw the tree after inserting key 16 using textbook `TREE-INSERT`.

**Type.** Trace CLRS `TREE-INSERT` (always inserts as a new leaf).

**Method (any instance).**
1. Walk down from the root: go left when `key < node.key`, right when `key > node.key`.
2. Stop at the empty (`NIL`) slot you reach; the new key becomes a leaf there.

**Worked solution.** Path for 16: `16 > 10` → right to 21; `16 < 21` → left to 15; `16 > 15` → right to 17; `16 < 17` → left of 17, which is empty. So **16 becomes the left child of 17**.
```
            10
          /    \
         5      21
        /      /  \
       3     15    30
            /  \
          11    17
               /
             16
```

## Opgave 2 (15%) — Huffman codes

### a (8%): Build the Huffman tree

**Problem.** Alphabet `{a,b,c,d,e,f}` with frequencies a=33, b=28, c=52, d=20, e=10, f=12. Draw the Huffman tree.

**Type.** Build a Huffman tree (greedy merge of two smallest).

**Method (any instance).**
1. Make one leaf per symbol with its frequency; put all in a min-priority queue.
2. Repeat until one tree remains: extract the two smallest-frequency nodes, make a new internal node whose two children are them, with frequency = sum, insert it back.
3. Label edges (left = 0, right = 1) for codes.

**Worked solution (merge order).**
- `e(10) + f(12) = 22`
- `d(20) + {e,f}(22) = 42`
- `b(28) + a(33) = 61`
- `{d,e,f}(42) + c(52) = 94`
- `{b,a}(61) + {{d,e,f},c}(94) = 155` (root)

One valid Huffman tree (children order is free; one consistent choice):
```
                 (155)
            0 /        \ 1
          (61)         (94)
         0/   \1      0/    \1
        b(28) a(33) (42)    c(52)
                   0/   \1
                 d(20)  (22)
                       0/   \1
                     e(10)  f(12)
```
Codes from this drawing: `b=00, a=01, d=100, e=1010, f=1011, c=11`. (checked: Python — merge order and code lengths a=2,b=2,c=2,d=3,e=4,f=4 confirmed.) (Any tree produced by the same merges and an arbitrary left/right convention is correct; the multiset of code lengths is what matters.)

### b (7%): Decode `1101001010101011`

**Problem.** Use the given Huffman tree (root: 0→a, 1→X; X: 0→Y, 1→f; Y: 0→P, 1→Q; P: 0→b, 1→c; Q: 0→d, 1→e) to decode the bit string `1101001010101011`.

**Type.** Decode a Huffman bit string (walk the tree).

**Method (any instance).**
1. Codes from the given tree: `a=0, f=11, b=1000, c=1001, d=1010, e=1011`.
2. Start at the root, follow each bit (0=left, 1=right). On reaching a leaf, output its symbol and return to the root. Repeat.

**Worked solution.** Split the string:
`11 | 0 | 1001 | 0 | 1010 | 1011` → `f, a, c, a, d, e`. Decoded text: **`facade`** (checked: Python, no leftover bits).

## Opgave 3 (15%) — Asymptotic notation

### a (15%): Which of the five statements are true?

**Problem.** Decide truth of:
1. `n² ∈ Ω(n)`
2. `n ∈ Θ(n²)`
3. `n log n ∈ o(n²)`
4. `log n ∈ O(√n)`
5. `n! ∈ ω(2ⁿ)`

**Type.** Classify growth-rate claims using limits.

**Method (any instance).** Use limit tests:
- `f ∈ O(g)`: `lim f/g < ∞`. `f ∈ o(g)`: `lim f/g = 0`. `f ∈ Ω(g)`: `lim f/g > 0` (possibly ∞). `f ∈ ω(g)`: `lim f/g = ∞`. `f ∈ Θ(g)`: `0 < lim f/g < ∞`.

**Worked solution.**
1. **True.** `n²/n = n → ∞`, so `n²` grows at least as fast as `n`; `n² ∈ Ω(n)`.
2. **False.** `n/n² = 1/n → 0`, so `n ∈ o(n²)`, hence `n ∉ Θ(n²)`.
3. **True.** `(n log n)/n² = (log n)/n → 0`, so `n log n ∈ o(n²)`.
4. **True.** `(log n)/√n → 0`, so `log n ∈ O(√n)` (even `o(√n)`). Numerically `log₂n/√n` falls 1.05 → 0.0027 as `n: 10 → 10⁸`.
5. **True.** `n!/2ⁿ → ∞` (each new factor `k` eventually exceeds 2), so `n! ∈ ω(2ⁿ)`. Numerically `20!/2²⁰ ≈ 2.3·10¹²`.

**Answer: statements 1, 3, 4, 5 are true; statement 2 is false.** (checked: Python — the relevant limit ratios.)

## Opgave 4 (20%) — Bipartite graphs

A bipartite graph (todelt graf) is one whose vertex set splits into disjoint `X`, `Y` so every edge has one endpoint in `X` and one in `Y`.

### a (6%): Which of the three graphs are bipartite? Give a 2-partition.

**Problem.** (a) a 5-cycle (pentagon); (b) a 4-cycle (square); (c) 5 vertices (top `T`, mid-left `L`, mid-right `R`, bottom-left `BL`, bottom-right `BR`) with edges `T–BR`, `L–R`, `L–BR`, `BL–R`, `BL–BR` (two crossing diagonals plus the top vertex hanging off `BR`).

**Type.** Recognize bipartiteness; a graph is bipartite iff it has no odd cycle.

**Method (any instance).** Try a 2-coloring: pick a start vertex, color it `X`, color all neighbors `Y`, their neighbors `X`, etc. If two adjacent vertices ever get the same color you hit an odd cycle → not bipartite.

**Worked solution.**
- **(a) pentagon = `C₅`:** odd cycle of length 5 → **not bipartite**. (2-coloring fails: tried, conflict appears.)
- **(b) square = `C₄`:** even cycle → **bipartite**. Partition `X = {top-left, bottom-right}`, `Y = {top-right, bottom-left}` (the two diagonal pairs).
- **(c) bipartite** (checked: Python). Reading the edges off the drawing: `T` only touches `BR` (degree 1), and `L, R, BL, BR` form a 4-cycle `L–R–BL–BR–L` (the edges `L–R`, `R–BL`, `BL–BR`, `BR–L`, no chords). A 4-cycle is even and a pendant vertex never creates an odd cycle, so the whole graph 2-colors. Partition: `X = {T, L, BL}`, `Y = {R, BR}` — every one of the five edges crosses between `X` and `Y`.

### b (7%): Modify BFS to decide bipartiteness, same asymptotic time

**Type.** Algorithm modification + correctness argument (no formal proof).

**Method / algorithm.** Run BFS but give each vertex a *level* parity (its BFS distance mod 2) used as its color. Concretely:
```
BIPARTITE-BFS(G):
  color all vertices "uncolored"
  for each vertex s:                 # handle disconnected graphs
    if s uncolored:
      color[s] = 0; d[s] = 0; enqueue s
      while queue not empty:
        u = dequeue
        for each neighbor v of u:
          if v uncolored:
            color[v] = 1 - color[u]   # opposite color of u
            enqueue v
          else if color[v] == color[u]:
            return FALSE              # same-color edge found
  return TRUE
```
**Running time.** Identical to BFS: each vertex enqueued once, each edge examined O(1) times → `O(V + E)`. The added parity test and color comparison are O(1) per edge.

**Correctness (informal).** BFS assigns each vertex a level = shortest-path distance from its source. Coloring by level parity is the only candidate 2-coloring consistent with a BFS tree, because every tree edge connects consecutive levels (different parity). The graph is bipartite iff this coloring is proper, i.e. no edge joins two same-parity vertices. A *non-tree* edge `(u,v)` with `color[u]==color[v]` connects two vertices of equal BFS distance; together with the tree paths to their lowest common ancestor it closes an odd cycle, so the graph is not bipartite — the algorithm correctly returns FALSE. If no such edge exists, the level-parity coloring is a valid 2-partition (`X` = even levels, `Y` = odd levels), so the graph is bipartite and the algorithm returns TRUE.

### c (7%): Prove: a graph is bipartite ⟺ it has no odd cycle (ulige kreds)

**Type.** Iff characterization argument.

**Worked solution.**

*(⇒) Bipartite ⟹ no odd cycle.* Let `X, Y` be the partition. Walk along any cycle `v₀, v₁, …, v_k = v₀`. Every edge crosses between `X` and `Y`, so the side alternates `X, Y, X, Y, …` at each step. To return to the starting side after `k` steps, `k` must be even. So every cycle has even length; there is no odd cycle.

*(⇐) No odd cycle ⟹ bipartite.* Assume no odd cycle. Work per connected component. Pick any vertex `s`, run BFS, and color each vertex by its distance parity from `s` (`X` = even distance, `Y` = odd). Suppose some edge `(u,v)` joins two same-parity vertices. Let `w` be the lowest common ancestor of `u` and `v` in the BFS tree. The tree path `w→u` and `w→v` have lengths of equal parity (since `dist(u)` and `dist(v)` have equal parity and both share the prefix to `w`), so their combined length is even; adding edge `(u,v)` closes a cycle of odd length — contradiction. Hence no edge joins same-parity vertices, so the parity coloring is a proper 2-partition and the component is bipartite. All components bipartite ⟹ the graph is bipartite.

## Opgave 5 (30%) — Parsimony on a binary tree

Input: a binary tree where every node has exactly 0 or 2 children; each leaf holds a character from alphabet `Σ`. Assign characters to internal nodes minimizing the number of *transition edges* (overgangskanter) — edges whose two endpoints hold different characters.

### a (5%): Assign characters minimizing transition edges

**Problem.** Tree (labels `x1..x9` from p. 9): root `x1`→(`x2`,`x3`); `x2`→(`x4`, leaf `x5=C`); `x3`→(leaf `x6=A`, leaf `x7=C`); `x4`→(leaf `x8=A`, leaf `x9=B`). Internal nodes: `x1,x2,x3,x4`. `Σ = {A,B,C}`.

**Type.** Solve a small instance of the parsimony / small-phylogeny problem by hand (Fitch / Sankoff).

**Method (any instance).** For each node compute `Cost(k,a)` = minimum transitions in `k`'s subtree if `k` is labeled `a` (bottom up). Leaf: 0 if its character is `a`, else `∞`. Internal: for each `a`, `Cost(k,a) = min_b[trans(a,b)+Cost(left,b)] + min_b[trans(a,b)+Cost(right,b)]`. Answer = `min_a Cost(root,a)`. Recover labels top-down by choosing minimizing characters.

**Worked solution.** Brute force over all `3⁴` internal labelings gives **minimum = 3 transition edges** (checked: Python). Optimal labelings (4 ties): `AAAA`, `CCCA`, `CCCB`, `CCCC` (writing `x1 x2 x3 x4`).

A clean optimal assignment: label every internal node **C** (`x1=x2=x3=x4=C`). Edges, with leaf values `x5=C, x6=A, x7=C, x8=A, x9=B`:
- `x1–x2 (C/C)`, `x1–x3 (C/C)`, `x2–x4 (C/C)`, `x2–x5 (C/C)`, `x3–x7 (C/C)`: same, 0.
- `x3–x6 (C/A)`, `x4–x8 (C/A)`, `x4–x9 (C/B)`: differ, 3 transition edges.

So 3 transitions, matching the brute-force minimum.

### b (7%): How many times is `Cost` called with `x4` as an argument?

**Problem.** With `Σ = {A,B,C}`, count calls `Cost(x4, ·)` when `TotalCost(x1)` runs the given recursive pseudocode.

**Type.** Count recursive calls of a naive recursion.

**Method (any instance).** Track how calls fan out:
- `TotalCost(root)` runs `Min1(root)`: calls `Cost(root, a)` once per `a` → `|Σ|` calls on the root.
- Each call `Cost(k, a)` on an internal node runs `Min2(k.left, a)` and `Min2(k.right, a)`; each `Min2` loops over all `b ∈ Σ`, calling `Cost(child, b)` `|Σ|` times.
- So calls on a child = `|Σ| ·` (calls on its parent). For a node at depth `d` (root depth 0): **calls = `|Σ|^{d+1}`**.

**Worked solution.** `x4` is at depth 2 (`x1`→`x2`→`x4`). Calls `= |Σ|^{2+1} = 3³ = ` **27** (checked: Python — call counts x1=3, x2=9, x4=27).

### c (8%): Argue `TotalCost` runs in `Ω(|Σ|^{log₂ n})`

**Problem.** Show the naive algorithm takes time `Ω(|Σ|^{log₂ n})`, `n` = number of nodes.

**Type.** Lower-bound argument on recursion blow-up.

**Worked solution.** From (b), a node at depth `d` receives `|Σ|^{d+1}` calls to `Cost`, each doing `≥ 1` work. For the worst case take a *balanced* full binary tree of `n` nodes: its height is `h = Θ(log₂ n)`, and it has leaves at depth `h`. Those deepest nodes alone receive `|Σ|^{h+1} ≥ |Σ|^{log₂ n}` calls. Total running time is at least the work at that level:
`T(n) ≥ |Σ|^{h+1} ≥ |Σ|^{log₂ n}`, i.e. `T(n) ∈ Ω(|Σ|^{log₂ n})`.
Note `|Σ|^{log₂ n} = n^{log₂ |Σ|}` — polynomial in `n` but with an exponent that grows with `|Σ|` (e.g. `|Σ|=4 ⇒ n²`). The blow-up comes from recomputing the same `Cost(k,a)` subproblem exponentially many times.

### d (10%): Rewrite `TotalCost` with dynamic programming; give time and space

**Problem.** Give a DP version. Each node has a unique id `k.id ∈ {1..n}`, and `Σ = {1..|Σ|}`.

**Type.** Convert recursion to DP (memoization / bottom-up table).

**Method / algorithm.** The recursion recomputes each `Cost(k,a)` many times, but there are only `n·|Σ|` distinct subproblems. Store them in a table `C[1..n][1..|Σ|]` and fill bottom-up (children before parents — e.g. post-order, or process ids so children precede parents).
```
PARSIMONY-DP(T):
  allocate C[1..n][1..|Σ|]
  for each node k in post-order (leaves first):
    if k is a leaf:
      for a = 1..|Σ|:
        C[k.id][a] = 0 if k.char == a else ∞
    else:
      L = k.left.id;  R = k.right.id
      for a = 1..|Σ|:
        minL = min over b of ( trans(a,b) + C[L][b] )   # trans = 0 if a==b else 1
        minR = min over b of ( trans(a,b) + C[R][b] )
        C[k.id][a] = minL + minR
  return min over a of C[root.id][a]
```

**Running time.** Each internal node computes `|Σ|` entries, and each entry takes a `min` over `|Σ|` values for the left child and `|Σ|` for the right → `Θ(|Σ|²)` per node. With `n` nodes: **`Θ(n · |Σ|²)`** worst case. Leaves cost `Θ(|Σ|)` each, dominated by this.

**Space.** The table is `n · |Σ|` entries → **`Θ(n · |Σ|)`**. (Could be reduced to `O(|Σ| · height)` if you discard children's rows after use, but `Θ(n·|Σ|)` is the straightforward bound.)
