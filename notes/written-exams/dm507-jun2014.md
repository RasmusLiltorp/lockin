## Opgave 1 (8%) — Solve recurrences

**Problem.** Give the solutions to:
- i) `T(n) = 2·T(n/3) + n`
- ii) `T(n) = 32·T(n/4) + n^2.5`

**Type.** Master theorem (lærebogens Theorem 4.1). Compare `f(n)` to `n^{log_b a}`.

**Method (any `T(n)=a·T(n/b)+f(n)`):**
1. Compute the critical exponent `c = log_b a`.
2. Compare `f(n)` with `n^c`:
   - `f` polynomially smaller (`f = O(n^{c-ε})`) → Case 1 → `Θ(n^c)`.
   - `f = Θ(n^c)` (possibly times `log^k`) → Case 2 → `Θ(n^c log n)` (k=0).
   - `f` polynomially larger and regularity holds → Case 3 → `Θ(f(n))`.

**Worked solution.**
- i) `a=2, b=3`, `c = log_3 2 ≈ 0.631`. `f(n)=n = n^1`, and `1 > 0.631` by a constant, so Case 3. Regularity `2·(n/3) = (2/3)n ≤ c'·n` holds. **`T(n) = Θ(n)`.**
- ii) `a=32, b=4`, `c = log_4 32 = 5/2 = 2.5`. `f(n)=n^2.5 = n^c`, Case 2. **`T(n) = Θ(n^{2.5} log n)`.**

## Opgave 2 (10%) — True/false asymptotics

**Problem.** State true/false for each.

**Type.** Definitions of O, Θ, o, ω.

**Method.** `f=O(g)`: f grows ≤ g up to constant. `f=Θ(g)`: same order. `f=o(g)`: strictly smaller (`f/g→0`). `f=ω(g)`: strictly larger (`f/g→∞`). For sums of powers, the highest power dominates.

**Worked solution.**

| # | Claim | Verdict | Why |
|---|-------|---------|-----|
| i | n² is O(n²) | **True** | equal order |
| ii | n² is Θ(n²) | **True** | equal order |
| iii | n⁴ is O(5n³+3n⁵) | **True** | RHS ~ n⁵ ≥ n⁴ |
| iv | n⁴ is Θ(5n³+3n⁵) | **False** | RHS is Θ(n⁵), n⁴ smaller |
| v | n log n is O(n^1.5) | **True** | log n = o(n^0.5) |
| vi | n is O(log n) | **False** | n grows faster |
| vii | (log n)^10 is O(n^0.10) | **True** | any polylog = o(any positive power) |
| viii | 1 is O(n) | **True** | constant ≤ n |
| ix | n² is o(n³) | **True** | n²/n³ = 1/n → 0 |
| x | n³ is ω(n³) | **False** | same order, ratio →1 not ∞ |

## Opgave 3 (6%) — Build-Max-Heap

**Problem.** Array `[5,4,3,2,1,10,9,8,7,6]` (1-indexed). Give the array after `Build-Max-Heap`.

**Type.** Trace Build-Max-Heap.

**Method.** For `i = ⌊n/2⌋` down to `1`, call `Max-Heapify(A,i)`. Each Max-Heapify sifts `A[i]` down, swapping with the larger child while it is bigger.

**Worked solution.** n=10, start at i=5 down to 1 (0-indexed i=4..0). Result verified:

**`[10, 8, 9, 7, 6, 3, 5, 2, 4, 1]`**

(left→right). Check root 10 ≥ children 8,9; node 8 ≥ 7,6; node 9 ≥ 3,5; etc.

## Opgave 4 (6%) — Double hashing

**Problem.** Table size 13. `h1(x)=(5x+1) mod 13`, `h2(x)=1+(x mod 12)`. Probe sequence `h(x,i)=(h1(x)+i·h2(x)) mod 13`. Starting table (index : value): 0:18, 2:8, 5:6, 8:30, 9:25, 11:2, 12:23 (rest empty). Insert 3, 5, 15 in that order; give the final table.

**Type.** Trace open-addressing double hashing.

**Method.** For each key, compute h1, h2; try `i=0,1,2,…` at index `(h1+i·h2) mod 13` until an empty slot; place key there.

**Worked solution.**
- **3:** h1=(16) mod13=3, h2=1+3=4. i=0 → idx 3 empty → place at **3**.
- **5:** h1=(26) mod13=0, h2=1+5=6. i=0 → idx 0 (18, taken); i=1 → (0+6)=6 empty → place at **6**.
- **15:** h1=(76) mod13=11, h2=1+(15 mod12)=1+3=4. Probe idx 11(2, taken), 11+4=15 mod13=2(8, taken), 11+8=19 mod13=6(now 5, taken), 11+12=23 mod13=10 empty → place at **10**.

Final table (index 0..12), empties = x:

**`18 x 8 3 x 6 5 x 30 25 15 2 23`**

## Opgave 5 (6%) — Radix-Sort

**Problem.** `A = [8345, 7112, 1830, 5001, 4345, 2222, 9112, 6363]`, run `Radix-Sort(A,4)` (4 digits, base 10, least-significant first, stable). Show A after three of the four iterations.

**Type.** Trace LSD radix sort.

**Method.** Pass `d=0` (ones) to `d=3` (thousands). Each pass = stable counting sort on that digit, preserving prior order for ties.

**Worked solution.**
- After iter 1 (ones digit): `[1830, 5001, 7112, 2222, 9112, 6363, 8345, 4345]`
- After iter 2 (tens digit): `[5001, 7112, 9112, 2222, 1830, 8345, 4345, 6363]`
- After iter 3 (hundreds digit): `[5001, 7112, 9112, 2222, 8345, 4345, 6363, 1830]`
- (After iter 4 (thousands), fully sorted: `[1830, 2222, 4345, 5001, 6363, 7112, 8345, 9112]`)

Any three of the four shown above answer the question; the three intermediate states are the natural choice.

## Opgave 6 (6%) — Huffman coding

**Problem.** Frequencies: a:400, e:750, i:300, o:150, u:200, y:100 (1900 total). Build a Huffman tree; give each codeword and the total encoded length.

**Type.** Build Huffman tree, read off codes.

**Method.** Repeatedly pop the two smallest-frequency nodes, merge into a node with summed frequency, push back. Continue to one root. Label left=0, right=1 down to leaves. (Codes depend on tie-breaking and 0/1 labeling; lengths are canonical.)

**Worked solution.** Merge order: (100+150=250 from y,o) → then 200(u)+250=450 → 300(i)+400(a)=700 → 450+700=1150 → 750(e)+1150=1900. One valid code set (verified):

| char | freq | code | len |
|------|------|------|-----|
| e | 750 | 0 | 1 |
| u | 200 | 100 | 3 |
| y | 100 | 1010 | 4 |
| o | 150 | 1011 | 4 |
| i | 300 | 110 | 3 |
| a | 400 | 111 | 3 |

**Total bits** = 750·1 + 200·3 + 100·4 + 150·4 + 300·3 + 400·3 = **4450 bits**.

Codeword lengths (1,3,4,4,3,3 for e,u,y,o,i,a) are forced by the tree; the exact 0/1 strings can differ by symmetric swaps but the total length is always **4450**.

## Opgave 7 (17%) — Kruskal + union-find

**Problem.** Graph G with edges (weights):

The first 7 examined edges and what happens:

2. (a,f)3 → add
3. (a,c)5 → add (joins {a,f} and {b,c})
4. (d,e)8 → add
5. (b,f)11 → cycle, skip
6. (d,f)12 → add (joins {a,b,c,f} and {d,e})

## Opgave 8 (10%) — Sorting 0/1 keys

**Problem.** Sort n elements whose keys are only 0 or 1. For CountingSort, InsertionSort, MergeSort, QuickSort give worst-case and best-case from: A) O(n), B) O(n log n), C) O(n²).

**Type.** Recall complexity, specialized to a 2-value key domain.

**Method.** Reason per algorithm with this input. Counting sort: range k=2 constant → Θ(n) always. Insertion sort: best when already sorted (O(n)); worst when many inversions. Merge sort: Θ(n log n) regardless of input. Quicksort: depends on partition balance; with only two key values and the standard last-element pivot, partitioning can degrade.

**Worked solution.**

| | Worst case | Best case |
|---|---|---|
| CountingSort | **A** O(n) | **A** O(n) |
| InsertionSort | **C** O(n²) | **A** O(n) |
| MergeSort | **B** O(n log n) | **B** O(n log n) |
| QuickSort | **C** O(n²) | **B** O(n log n) |

Notes: InsertionSort best case = already-sorted 0…0 1…1 (O(n)); worst = e.g. all 1s then 0s giving Θ(n²) shifts. QuickSort worst case occurs because with only 0/1 keys a Lomuto/last-element pivot produces extremely unbalanced (size 0 vs n−1) splits on inputs like all-equal keys → Θ(n²); a balanced split gives the O(n log n) best case. (Counting sort is the right tool: O(n).)

## Opgave 9 (16%) — Loop-nest runtimes

**Problem.** Give asymptotic O-running time as a function of n for four pseudocode fragments.

**Type.** Count loop iterations.

**Method.** Multiply/sum the iteration counts of nested loops; a `while` that halves a value runs Θ(log) times; a halving outer loop with linear inner work sums to a geometric series.

**Worked solution.**
- **Algoritme1:** `for i=1..n for j=i..n s++`. Iterations = Σ_{i=1}^n (n−i+1) = n(n+1)/2 → **O(n²)**.
- **Algoritme2:** `for i=1..n { s=n; while s>1 s=⌊s/2⌋ }`. Inner while runs ⌊log₂ n⌋ times, outer n times → **O(n log n)**.
- **Algoritme3:** `for i=1..n for j=i..n for k=i..j s++`. Triple nest bounded by n each, count Σ_i Σ_{j≥i} (j−i+1) = Θ(n³) → **O(n³)**. (Verified count grows like n³/6.)
- **Algoritme4:** `while n>1 { for i=1..n s++; n=⌊n/2⌋ }`. Work = n + n/2 + n/4 + … ≈ 2n → **O(n)**.

## Opgave 10 (15%) — Square graph shortest paths

**Problem.** A "square graph" (kvadrat-graf) is a directed graph with k rows of k nodes each. Nodes `v_{i,j}`, i=row 1..k, j=col 1..k. Edges: `(v_{i,j}, v_{i+1,j})` (up one row), `(v_{i,j}, v_{i,j+1})` (right), `(v_{i,j}, v_{i,j-1})` (left), whenever both endpoints exist. All edge weights are non-negative.

**a (3%) — n and m as functions of k.**
- Nodes: **n = k²**.
- Edges: vertical "up" edges k(k−1); horizontal: each row has (k−1) right + (k−1) left = 2(k−1), over k rows = 2k(k−1). Total **m = 3k(k−1)** = 3k² − 3k. (Verified: k=5 → n=25, m=60.)

**b (4%) — Dijkstra running time on this graph from v_{1,1}.**
With a binary-heap priority queue Dijkstra is O((n+m) log n). Here n=k², m=Θ(k²), so **O(k² log k)** (since log n = log k² = 2 log k = Θ(log k)).

**c (8%) — O(m) algorithm for shortest paths from v_{1,1} to all nodes.**

**Type.** Design + correctness + runtime (uses the structure: edges only go up a row or sideways within a row, never down).

**Key observation.** Every edge stays in the same row (left/right) or goes to the next row up (i→i+1). So no path ever decreases its row index. Processing rows bottom-to-top (i = 1,2,…,k) is a valid topological order across rows: once we finalize row i, no later relaxation can come back to it. (This is the "process vertices in an order where all predecessors are done" idea, cf. the DAG-shortest-path Lemma 24.15.)

**Algorithm.**
```
dist[v_{1,1}] = 0; all other dist = +inf
for i = 1 to k:                       # rows bottom to top
    # 1) settle horizontal edges inside row i (it is a path graph)
    for j = 2 to k:                   # left-to-right sweep
        relax edge (v_{i,j-1}, v_{i,j})
    for j = k-1 down to 1:            # right-to-left sweep
        relax edge (v_{i,j+1}, v_{i,j})
    # 2) push distances up to row i+1
    if i < k:
        for j = 1 to k:
            relax edge (v_{i,j}, v_{i+1,j})
```
where `relax(u,v): if dist[u]+w(u,v) < dist[v]: dist[v] = dist[u]+w(u,v)`.

**Why two horizontal sweeps suffice per row.** Within a row, distances enter at some set of columns (from the row below, plus the source in row 1). The row is a bidirectional path. For a path graph with non-negative weights, the shortest distance to every node from a set of pre-seeded entry points is obtained by one left-to-right relaxation pass followed by one right-to-left pass: the left-to-right pass propagates every "comes from the left" contribution, the right-to-left pass propagates every "comes from the right" contribution. Because all weights are ≥ 0, an optimal path inside the row never reverses horizontal direction, so two monotone sweeps capture all optimal in-row paths.

**Correctness.** By induction on rows. Before processing row i, every entry distance into row i (from completed row i−1, and the source for row 1) is already correct, because the only edges into row i come from row i−1 (up edges) — handled in step 2 of iteration i−1 — and from inside row i itself. The two sweeps then compute correct shortest distances for all of row i (path-graph argument above). No edge ever leads into a lower-numbered row, so finalized rows are never invalidated. Hence after iteration k all dist[] are correct. (Verified against Dijkstra on 200 random square graphs, k=2..6, exact match.)

**Running time.** Each iteration does O(k) work (three linear sweeps over a row of k nodes); k iterations → **O(k²) = O(m)** since m = Θ(k²). Initialization is O(n)=O(k²) too. So the algorithm is **O(m)**.
