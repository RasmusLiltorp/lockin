## Opgave 1 — Red-black trees (18%)

### 1a (8%)

**Problem.** Figure 1 shows eight trees T1–T8. Bold nodes are black, the rest red; the small circles are the black NIL leaves. State which of the eight are valid red-black trees (rød-sorte træer).

**Type.** Check the red-black properties on each tree.

**Method (any RB-validity check).**
1. Root is black.
2. Every leaf (NIL) is black — given by the drawing.
3. No red node has a red child (no red-red edge).
4. Every root-to-NIL path through a node has the same number of black nodes (equal black-height). Easiest test: at each node, the black-height seen down the left subtree must equal the one down the right; an empty (NIL) side counts as black-height 1.
5. Valid iff all four hold. Swap in your own colored tree.

**Worked solution.** Reading the figure (key, color):

- **T1**: root 1(B); children 2(B), 3(B). Root black, no red-red, all paths have the same blacks. **Valid.**
- **T2**: root 2(R). Root is red → property 1 fails. **Invalid.**
- **T3**: root 2(B); children 1(R), 3(R). Black root, red leaves with black NIL children, equal black-height. **Valid.**
- **T4**: root 2(B); children 1(B), 3(B). **Valid.**
- **T5**: 2(B); 1(R); 3(R) with right child 4(R). Red 3 has red child 4 → red-red. **Invalid.**
- **T6**: 2(B); 1(R); 3(R) with right child 4(B). Under node 3 the NIL-left path has black-height 1 but the path through 4(B) has black-height 2 → property 4 fails. **Invalid.**
- **T7**: 2(B); 1(B); 4(R) with left child 3(B). Under node 4 the path through 3(B) has black-height 2 but the right NIL has black-height 1 → property 4 fails. **Invalid.**
- **T8**: 2(B); 1(B); 4(B) with left child 3(R). Black root, no red-red (3 red under black 4), and black-heights match (left side 1(B)→NIL = 2 blacks; right side 4(B)→{3(R) or NIL} = 2 blacks). **Valid.**

**Answer: T1, T3, T4, T8 are red-black trees.** (T2, T5, T6, T7 are not.) (checked: Python — black-height + red-red + root checks on all eight)

### 1b (10%)

**Problem.** Insert key 3 into the red-black tree T below. Which tree in Figures 2–3 (T1–T10) equals T after the insertion?

T:
```
            7(B)
        5(R)        8(B)
   2(B)   6(B)          9(R)
 1(R) 4(R)
```

**Type.** Trace RB-INSERT (CLRS) plus fixup.

**Method (any RB insertion).**
1. Do a normal BST insert; color the new node red.
2. While the parent is red, run the fixup looking at the uncle:
   - **Case 1** (uncle red): recolor parent and uncle black, grandparent red; move the pointer up to the grandparent.
   - **Case 2** (uncle black, node is the "inner" child): rotate at the parent to turn it into case 3.
   - **Case 3** (uncle black, node is the "outer" child): recolor parent black and grandparent red, then rotate at the grandparent.
3. Color the root black.
4. Compare against the candidate trees. Swap in your own key/tree.

**Worked solution.**
- BST insert: 3 < 7 → left; 3 < 5 → left; 3 > 2 → right to 4; 3 < 4 → left child of 4. New node **3(R)**.
- Parent 4 is red. Grandparent 2, uncle = 1 (left child of 2). 1 is **red** → **Case 1**: color 1 and 4 black, color 2 red; move up to z = 2.
- Now z = 2(R), parent 5(R), grandparent 7(B), uncle 8 is **black**. z = 2 is left child and 5 is left child → "left-left", **Case 3**: color 5 black, color 7 red, right-rotate at 7.
- After the right-rotation 5 becomes the root; 7 becomes 5's right child and takes 6 as its left child.
- Root recolored black (already black).

Result:
```
            5(B)
       2(R)        7(R)
   1(B) 4(B)    6(B)  8(B)
        3(R)              9(R)
```

This is exactly **T7** (in Figure 3): root 5(B); left subtree 2(R) with children 1(B) and 4(B), and 3(R) as left child of 4; right subtree 7(R) with children 6(B) and 8(B), and 9(R) as right child of 8.

**Answer: T7.** (checked: Python — full CLRS RB-INSERT with rotations reproduces this exact tree)

## Opgave 2 — Asymptotic notation (15%)

**Problem.** Let f1, f2, g1, g2 be positive functions with f1 ∈ O(g1) and f2 ∈ O(g2). Which of these are always true?
- (a) f1(n) + f2(n) ∈ O(g1(n) + g2(n))
- (b) g1(n) + g2(n) ∈ Ω(f1(n) + f2(n))
- (c) f1(n)/f2(n) ∈ O(g1(n)/g2(n))

**Type.** Prove or give counterexample for O/Ω closure rules.

**Method (any "is this asymptotic rule valid" question).**
1. Write the definitions: f ∈ O(g) means f(n) ≤ c·g(n) for some c > 0 and all large n.
2. For a claimed rule, try to derive it from the hypotheses with explicit constants.
3. If it does not go through, build a small counterexample (pick concrete simple functions for f1, f2, g1, g2 satisfying the hypotheses but breaking the conclusion).

**Worked solution.**
- **(a) True.** f1 ≤ c1·g1 and f2 ≤ c2·g2 for large n. Then f1 + f2 ≤ c1·g1 + c2·g2 ≤ max(c1,c2)·(g1+g2). With c = max(c1,c2) this is the O-bound. (All functions positive, so the sum on the right is positive.)
- **(b) True.** It is just (a) restated. (a) gives f1+f2 ≤ c(g1+g2), i.e. g1+g2 ≥ (1/c)(f1+f2), which is exactly g1+g2 ∈ Ω(f1+f2). In general f ∈ O(g) ⟺ g ∈ Ω(f).
- **(c) False.** Counterexample: f1(n)=n, g1(n)=n (so f1∈O(g1)); f2(n)=1, g2(n)=n (so f2∈O(g2), since 1≤n). Then f1/f2 = n, but g1/g2 = n/n = 1. n ∉ O(1), so the claim fails. (Checked numerically: the ratio (f1/f2)/(g1/g2) = n grows without bound.)

**Answer: (a) true, (b) true, (c) false.**

## Opgave 3 — Sorting in two special cases (24%)

**Setup.** n keys, integers in the range 0 … n^5. Two inputs:
1. all keys distinct, reverse-sorted (descending), e.g. {12, 9, 8, 5, 2};
2. all keys equal, e.g. {5, 5, 5, 5, 5}.

For each algorithm give the running time in each case.

**Type.** State running times from the known behavior of each sort on a specific input pattern.

**Method (any "running time on this input" question).**
1. Recall the algorithm's cost as a function of comparisons/moves driven by the input order.
2. Reverse-sorted: this is the worst case for comparison sorts that exploit existing order (Insertion Sort: every element shifts all the way; Quicksort with first/last pivot: maximally unbalanced splits).
3. All-equal keys: count comparisons when every key ties.
4. For Radix Sort use d·(n + base); pick the base and digit count from the key range (here 0 … n^5).

### 3a — Insertion Sort
- **Reverse-sorted (case 1):** worst case. Each new key is smaller than all sorted ones, so it shifts past all of them: Θ(n²).
- **All equal (case 2):** with the standard `while A[i] > key` test, the new key is never strictly greater, so the inner loop never moves anything — one comparison per element. **Θ(n)**.

### 3b — Quicksort (textbook, last element as pivot, no randomization)
- **Reverse-sorted (case 1):** the pivot is the minimum each time → splits of size 0 and n−1 → **Θ(n²)** worst case.
- **All equal (case 2):** PARTITION with all keys equal puts everything on one side → most unbalanced split again → **Θ(n²)**.
(If randomized-pivot Quicksort is assumed: case 1 becomes Θ(n log n) expected; case 2 stays Θ(n²) with the standard CLRS PARTITION, because equal keys still give a degenerate split.)

### 3c — Heapsort
- Heapsort is **Θ(n log n)** on every input. BUILD-MAX-HEAP is O(n) and the n extract-max steps cost O(log n) each regardless of order.
- **Reverse-sorted (case 1):** **Θ(n log n)**.
- **All equal (case 2):** **Θ(n log n)** (sift-down still walks the height; the bound does not improve to linear for the standard algorithm).

### 3d — Radix Sort (best achievable)
Keys are in 0 … n^5. Write each key in base n: a number < n^5 needs at most **5 digits** in base n (since n^5 in base n is "100000", and keys are ≤ n^5, so 6 digits in the extreme; treating the range as 0..n^5−1 gives exactly 5 base-n digits). Counting-sort each digit: each pass is O(n + n) = O(n), and there are d = 5 passes. Total **Θ(5·n) = Θ(n)** — and this is the same for both inputs, because Radix Sort's cost does not depend on the input order.
- **Reverse-sorted (case 1):** **Θ(n)**.
- **All equal (case 2):** **Θ(n)**.

(General recipe: with keys in 0 … n^k, choose base n, get d = k+1 digits, time Θ((k+1)·n) = Θ(n) for constant k.)

## Opgave 4 — Which shortest-path algorithm fits which graph (18%)

**Graphs (Figure 4), directed, weighted, source s, target t** (verified with Bellman-Ford):
- **G1**: s→a(1), a→b(1), s→b(1), b→t(1). All weights equal to 1; acyclic (DAG). dist(s,t)=2.
- **G2**: s→a(10), a→t(1), s→b(1), b→a(1), a→c(1), c→b(1). All weights ≥ 0; contains a cycle (a→c→b→a). dist(s,t)=3.
- **G3**: s→a(6), a→c(−3), s→c(10), s→b(2), b→c(2), c→t(1). Has a negative edge (−3); acyclic (DAG). dist(s,t)=4 (path s→a→c→t = 6−3+1).

**Type.** Check each algorithm's precondition against each graph.

**Method (any "can algorithm X compute the shortest s–t distance here" question).**
1. **BFS** computes shortest distances only when all edge weights are equal (it counts edges). Works iff every weight is the same.
2. **DAG-Shortest-Paths** requires the graph to be acyclic (it relaxes edges in topological order). Negative weights are fine.
3. **Dijkstra** requires all edge weights ≥ 0. Cycles are fine.
4. Tick each graph that satisfies the precondition.

### 4a — Breadth-First Search
Needs all weights equal.
- G1: all weights 1 → **yes**.
- G2: weights 10 and 1, not equal → **no**.
- G3: weights mixed (6, −3, …) → **no**.

**Answer: only G1.** (checked: Python — G1 is the only graph with all edge weights equal)

### 4b — DAG-Shortest-Paths
Needs the graph acyclic.
- G1: DAG → **yes**.
- G2: has cycle a→c→b→a → **no**.
- G3: DAG (negative edge is allowed) → **yes**.

**Answer: G1 and G3.** (checked: Python — only G2 has a cycle (a→c→b→a))

### 4c — Dijkstra
Needs all weights ≥ 0.
- G1: all ≥ 0 → **yes**.
- G2: all ≥ 0 (cycle is fine) → **yes**.
- G3: edge a→c = −3 < 0 → **no**.

**Answer: G1 and G2.** (checked: Python — only G3 has a negative edge (a→c = −3))

## Opgave 5 — Max-weight independent set in a rooted tree (25%)

**Setup.** Rooted tree T, each node has a positive integer weight. A set of nodes is *independent* (uafhængig) if it contains no two neighbors (if x is in the set, none of x's children and not x's parent are in it). Find the independent set of maximum total weight.

The DP per node x:
- `With[x]` = best weight of the subtree rooted at x **if x is included** = w[x] + Σ over children c of `Without[c]`.
- `Without[x]` = best weight of the subtree rooted at x **if x is excluded** = Σ over children c of max(`With[c]`, `Without[c]`).
- Answer for the whole tree = max(`With[root]`, `Without[root]`).

(checked: Python — against the figure-5b example T1, this DP reproduces the entire published table: With=[18,6,6,3,5,1,2,1,2,2,1], Without=[15,9,4,4,1,0,0,0,0,0,0].)

### 5a (10%) — Fill the table for T2 (Figure 6)

T2 structure (read from Figure 6): a(1) is root with children b(2), c(5), d(4). b has two children e(2) and f(3). c is a leaf. d has child g(2). e has child h(4). So f hangs under b, not under c.

Compute bottom-up. (checked: Python)
- Leaves c, f, g, h: With = own weight, Without = 0.
- c: With = 5, Without = 0. f: With = 3, Without = 0. g: With = 2, Without = 0. h: With = 4, Without = 0.
- e: With = w[e] + Without[h] = 2 + 0 = 2; Without = max(With[h], Without[h]) = max(4,0) = 4.
- b: With = w[b] + Without[e] + Without[f] = 2 + 4 + 0 = 6; Without = max(With[e],Without[e]) + max(With[f],Without[f]) = max(2,4) + max(3,0) = 4 + 3 = 7.
- d: With = w[d] + Without[g] = 4 + 0 = 4; Without = max(With[g], Without[g]) = max(2,0) = 2.
- a: With = w[a] + Without[b] + Without[c] + Without[d] = 1 + 7 + 0 + 2 = 10;
  Without = max(With[b],Without[b]) + max(With[c],Without[c]) + max(With[d],Without[d]) = max(6,7) + max(5,0) + max(4,2) = 7 + 5 + 4 = 16.

| x        | a  | b | c | d | e | f | g | h |
|----------|----|---|---|---|---|---|---|---|
| With x   | 10 | 6 | 5 | 4 | 2 | 3 | 2 | 4 |
| Without x| 16 | 7 | 0 | 2 | 4 | 0 | 0 | 0 |

### 5b (5%) — A maximum-weight independent set in T2

Answer at root = max(With[a]=10, Without[a]=16) = **16**, achieved by excluding a. Trace which children realize the max (checked: Python):
- For a (excluded): take the better option per child. b: max(6,7)=7 → exclude b. c: max(5,0)=5 → include c. d: max(4,2)=4 → include d.
- b excluded ⇒ each child of b may be in: e: max(2,4)=4 → exclude e; f: max(3,0)=3 → include f. e excluded ⇒ h may be in: include h.
- c included (leaf). d included ⇒ g excluded.

**Optimal set = {c, d, f, h}, total weight 5 + 4 + 3 + 4 = 16.** It is independent: c, d, f have no common edge in the set (their parents a, b are out), and h's parent e is out.

### 5c (10%) — DP algorithm, running time, space

**Type.** Give a dynamic-programming algorithm on a tree.

**Algorithm.** Compute `With[x]` and `Without[x]` in one post-order traversal (process all children before the node).

```
MAX-WEIGHT-IND-SET(T):
    DFS-DP(root)
    return max(With[root], Without[root])

DFS-DP(x):
    With[x]    = w[x]
    Without[x] = 0
    for each child c of x:
        DFS-DP(c)
        With[x]    += Without[c]
        Without[x] += max(With[c], Without[c])
```

**Why it is correct.** If x is in the set, none of its children may be → each child contributes its best "child excluded" value, Without[c]. If x is out, each child may be in or out → each contributes max(With[c], Without[c]). The subtrees are independent once the choice at x is fixed (optimal substructure), so the recurrence gives the optimum.

**Running time.** Each node is visited once and does O(1) work per child; the total number of (node, child) pairs is the number of edges = n − 1. So **Θ(n)** time, n = number of nodes.

**Space.** Two arrays With[·] and Without[·] of size n, plus the recursion stack of depth = tree height (O(n) worst case). **Θ(n)** space.
