### 1a â€” Solve the recurrence (5%)

**Problem.** Solve `T(n) = 16Â·T(n/2) + n^4 + n^2`.

**Type.** Solve a divide-and-conquer recurrence with the Master theorem.

**Method (any `T(n)=aÂ·T(n/b)+f(n)`).**
1. Read off `a`, `b`, `f(n)`. Compute the watershed exponent `c* = log_b a`, i.e. compare `f(n)` to `n^(log_b a)`.
2. Case 1: if `f(n)=O(n^{c*-Îµ})` for some `Îµ>0` â†’ `T(n)=Î˜(n^{c*})`.
3. Case 2: if `f(n)=Î˜(n^{c*})` (optionally `Â·log^k n`) â†’ `T(n)=Î˜(n^{c*}Â·log^{k+1} n)`.
4. Case 3: if `f(n)=Î©(n^{c*+Îµ})` and the regularity condition `aÂ·f(n/b)â‰¤dÂ·f(n)` (`d<1`) holds â†’ `T(n)=Î˜(f(n))`.

**Worked solution.** `a=16`, `b=2`, so `c* = log_2 16 = 4`, watershed `n^4`. The driving term `f(n)=n^4+n^2 = Î˜(n^4)`. That equals `n^{c*}` exactly, so Case 2 with `k=0`.

**Answer: `T(n) = Î˜(n^4 Â· log n)`.**

The `n^2` is dominated by `n^4` and does not change the class.

### 1b â€” Huffman tree (5%)

**Problem.** Give a Huffman tree for the alphabet with frequencies a=300, b=150, c=75, d=125, e=200, f=50, g=100.

**Type.** Build a Huffman code (greedy, repeatedly merge two smallest).

**Method.** Put all symbols in a min-priority queue keyed by frequency. Repeatedly extract the two smallest, make them children of a new internal node with summed weight, reinsert. Stop with one tree. A 0/1 label on each edge gives the codes. (Tree is not unique; tie-breaking is free.)

**Worked solution.** Merge order (extract two smallest each step):

| step | merge | new weight |
|---|---|---|
| 1 | f(50) + c(75) | 125 |
| 2 | g(100) + d(125) | 225 |
| 3 | {f,c}(125) + b(150) | 275 |
| 4 | e(200) + {g,d}(225) | 425 |
| 5 | {f,c,b}(275) + a(300) | 575 |
| 6 | {e,g,d}(425) + {f,c,b,a}(575) | 1000 |

One resulting code (left=0, right=1; depends on which child is 0):

| symbol | freq | code | length |
|---|---|---|---|
| a | 300 | 11 | 2 |
| e | 200 | 00 | 2 |
| b | 150 | 101 | 3 |
| d | 125 | 011 | 3 |
| g | 100 | 010 | 3 |
| c | 75 | 1001 | 4 |
| f | 50 | 1000 | 4 |

Weighted path length (total encoded bits) = `300Â·2 + 200Â·2 + 150Â·3 + 125Â·3 + 100Â·3 + 75Â·4 + 50Â·4 = 2625` bits. Any Huffman tree built from the same merges has WPL 2625; only the labelling/left-right choice varies.

## Opgave 3 (25%) â€” Binary-digit algorithm

```
BinaryDigits(n)
  i = 0
  d = n
  while d > 0
     b_i = d mod 2
     d   = d div 2
     i   = i + 1
  k = i - 1
```
Identity given: `x = yÂ·(x div y) + (x mod y)` for all integers `x,y`.

### 3a â€” Run on `n = 55` (6%)

### 3b â€” Loop invariant (9%)

**Problem.** Prove that at every entry to the while-test:
(i) `n = dÂ·2^i + Î£_{j=0}^{i-1} b_jÂ·2^j`, (ii) `d â‰¥ 0`.

**Type.** Loop-invariant proof (initialization / maintenance / termination), here init+maintenance.

**Initialization.** Before the first test, `i=0`, `d=n`. The sum is empty (=0), so RHS `= nÂ·2^0 + 0 = n`. (i) holds. `n` is a positive integer so `d=nâ‰¥0`; (ii) holds.

**Maintenance.** Assume (i),(ii) hold at a test with values `d, i` and we enter the body (so `d>0`). Let `d', i'` be the values at the next test. The body sets `b_i = d mod 2`, `d' = d div 2`, `i' = i+1`.
Use the identity with `x=d, y=2`: `d = 2Â·(d div 2) + (d mod 2) = 2Â·d' + b_i`. Then
`d'Â·2^{i'} + Î£_{j=0}^{i'-1} b_jÂ·2^j = d'Â·2^{i+1} + Î£_{j=0}^{i} b_j 2^j = (2d')Â·2^i + b_iÂ·2^i + Î£_{j=0}^{i-1} b_j 2^j = (2d' + b_i)Â·2^i + Î£_{j=0}^{i-1} b_j 2^j = dÂ·2^i + Î£_{j=0}^{i-1} b_j 2^j = n`
by the induction hypothesis (i). So (i) holds for `d', i'`. For (ii): `d>0` integer â‡’ `d' = d div 2 â‰¥ 0`. Done.

### 3c â€” Correctness (6%)

**Problem.** Show the algorithm computes `n`'s binary representation for every positive integer `n`.

**Termination.** `d` is a non-negative integer and strictly decreases each iteration: for `dâ‰¥1`, `d div 2 < d` (since `d div 2 â‰¤ d/2 < d`). A strictly decreasing sequence of non-negative integers is finite, so the loop ends with `d=0`.

**Result.** At loop exit, `d=0`, and let `i` be its final value (so `k=i-1`). The invariant (i) gives `n = 0Â·2^i + Î£_{j=0}^{i-1} b_jÂ·2^j = Î£_{j=0}^{k} b_jÂ·2^j`. Each `b_j = (something) mod 2 âˆˆ {0,1}`. So `b_k b_{k-1}â€¦b_0` is a binary representation of `n` using digits 0/1. It has no leading zero: the last digit produced, `b_k`, came from the final non-zero `d` (the iteration that drove `d` to 0 had `d=1` at entry, giving `b_k = 1 mod 2 = 1`). Hence the output is exactly `n`'s binary representation.

### 3d â€” Runtime (4%)

`d` starts at `n` and halves (integer division) each iteration, reaching 0. The number of iterations is `âŒŠlog_2 nâŒ‹ + 1 = Î˜(log n)`. Each iteration does O(1) work (one mod, one div, one add) â€” treating arithmetic on these numbers as unit cost. **Running time `Î˜(log n)`.**

## Opgave 4 (25%) â€” Weighted longest common subsequence (DP)

Weights `w(a)=2, w(b)=4, w(c)=1, w(d)=3`. `X=acbd` (n=4), `Y=bacda` (m=5).
Recurrence:
```
W(i,j) = 0                                            if i=0 or j=0
       = max{ W(i-1,j), W(i,j-1) }                    if i,jâ‰¥1 and x_i â‰  y_j
```

### 4b â€” DP algorithm and runtime (8%)

**Method / algorithm.**
1. Allocate `(n+1)Ã—(m+1)` table `W`. Set row 0 and column 0 to 0.
2. Fill in increasing `i`, then increasing `j`, using the recurrence (each cell reads only `W[i-1][j]`, `W[i][j-1]`, `W[i-1][j-1]`, all already filled).
3. Return `W[n][m]`.

```
WeightedLCS(X,Y,w):
  for i in 0..n: W[i][0] = 0
  for j in 0..m: W[0][j] = 0
  for i in 1..n:
    for j in 1..m:
      if X[i] == Y[j]:
        W[i][j] = max(W[i-1][j], W[i][j-1], W[i-1][j-1] + w(X[i]))
      else:
        W[i][j] = max(W[i-1][j], W[i][j-1])
  return W[n][m]
```

**Runtime.** `(n+1)(m+1)` cells, O(1) per cell â‡’ **`Î˜(nÂ·m)`** time and `Î˜(nÂ·m)` space (reducible to `Î˜(min(n,m))` if only the value is needed). Weight lookups `w(c)` are O(1).

### 4c â€” Correctness of the recurrence (8%)

**Type.** Justify an optimal-substructure recurrence.

Consider an optimal common subsequence `Z` of the prefixes `X_i = x_1â€¦x_i` and `Y_j = y_1â€¦y_j`, of value `W(i,j)`.

- **Base `i=0` or `j=0`.** One prefix is empty, the only common subsequence is empty, value 0.
- **Case `x_i â‰  y_j`.** `Z` cannot use both `x_i` and `y_j` as its last matched character (they differ). So at least one of the two characters is unused by `Z`:
  - If `x_i` is not used, `Z` is a common subsequence of `X_{i-1}` and `Y_j`, value `â‰¤ W(i-1,j)`.
  - If `y_j` is not used, `Z` is a common subsequence of `X_i` and `Y_{j-1}`, value `â‰¤ W(i,j-1)`.
  Hence `W(i,j) â‰¤ max{W(i-1,j), W(i,j-1)}`. Both candidates are themselves achievable common subsequences of `X_i,Y_j`, so equality holds.
- **Case `x_i = y_j = c`.** Either `Z` matches this last equal pair or it does not.
  - If it does: drop the last matched `c`; the rest is a common subsequence of `X_{i-1}, Y_{j-1}`, so `W(i,j) = W(i-1,j-1) + w(c)` is achievable, and any `Z` using the pair has value `â‰¤ W(i-1,j-1)+w(c)`.
  - If it does not match the pair: as in the unequal case, `Z` is covered by `W(i-1,j)` or `W(i,j-1)`.
  Taking the max of the three covers every possibility and each is achievable, so `W(i,j) = max{W(i-1,j), W(i,j-1), W(i-1,j-1)+w(c)}`.

Since weights are positive, the third term is the one that usually wins on a match, but keeping the other two in the max is needed because a longer non-matching alignment can outweigh a single match. The recurrence is therefore exact.

## Opgave 5 (15%) â€” Augmenting a BST with `maxGap`

Each node `v` stores key `v.key` plus: `v.maxGap` (largest gap between consecutive keys in `v`'s subtree), `v.max` (largest key in subtree), `v.min` (smallest key in subtree). For a single-key subtree `v.maxGap = 0`. Root's `r.maxGap` gives the global largest predecessor-gap in O(1).

### 5a â€” Compute a node's fields from its children in O(1) (6%)

**Type.** Define an augmentation that is a function of a node + its two children's stored fields (so it composes bottom-up â€” Theorem on augmenting RB-trees).

For a node `v` with left child `L` and right child `R` (either may be NIL):

```
v.min = (L != NIL) ? L.min : v.key
v.max = (R != NIL) ? R.max : v.key
```

For `v.maxGap`, the consecutive-key gaps inside `v`'s subtree are: all gaps already inside the left subtree, all gaps already inside the right subtree, plus the two new gaps that straddle `v`:
- between the largest key in the left subtree and `v.key`: `v.key - L.max`,
- between `v.key` and the smallest key in the right subtree: `R.min - v.key`.

```
v.maxGap = max(
    (L != NIL) ? L.maxGap : 0,
    (R != NIL) ? R.maxGap : 0,
    (L != NIL) ? v.key - L.max : -infinity,   // omit term if no left child
    (R != NIL) ? R.min - v.key : -infinity )  // omit term if no right child
```

Because the BST orders keys, `L.max` is the in-order predecessor of `v.key` and `R.min` is its successor, so these two straddling gaps are exactly the consecutive-key gaps that cross `v`. This reads only `v.key`, the children's keys, and the children's three fields â€” O(1).

### 5b â€” Maintain under insert/delete in RB-tree without changing O(log n) (4%)

`v.maxGap, v.max, v.min` each depend only on `v` and its two children (5a). By the augmentation theorem (CLRS), a field that can be recomputed in O(1) from a node and its children can be maintained during RB-tree insert and delete without changing the O(log n) bound:
- **Insert/delete proper** changes the tree only along one root-to-leaf path; recompute the fields bottom-up along that path, O(1) per node, O(log n) total.
- **Rotations** are local: a rotation changes the child structure of only the two rotated nodes, so recomputing their fields (children first, then parent) restores correctness in O(1) per rotation, and there are O(1) rotations per operation.

So insertions and deletions stay O(log n).

### 5c â€” Find a key achieving the maximum gap, O(log n) (5%)

We want a concrete key `x` whose gap to its predecessor equals `r.maxGap = G`. Start at the root and walk down, always moving into a subtree that still contains a gap of size `G`. At node `v` with children `L,R`, the value `v.maxGap` was the max of four quantities (5a). Check which one realized it:

```
SearchMaxGap(v):           // precondition: v.maxGap == G (the global max)
  if R != NIL and R.min - v.key == G:
     return v.key          // the gap is between R.min's... actually between v.key and R.min:
                           // its right endpoint key is R.min, predecessor v.key -> return R.min
  if L != NIL and v.key - L.max == G:
     return v.key          // gap between L.max (predecessor) and v.key -> return v.key
  if L != NIL and L.maxGap == G:
     v = L; recurse
  else // R != NIL and R.maxGap == G
     v = R; recurse
```

Precisely: if the straddling gap on the left side `v.key - L.max == G`, the key with that predecessor gap is `v.key` (predecessor `L.max`). If the straddling gap on the right `R.min - v.key == G`, the realizing key is `R.min` (its predecessor is `v.key`). Otherwise the max gap sits entirely inside the left subtree (`L.maxGap == G`) or the right subtree (`R.maxGap == G`); descend there and repeat. One of the four cases always matches because `v.maxGap = G` was a max of exactly those four. Each step goes down one level, O(1) work, so total **O(log n)** in an RB-tree.
