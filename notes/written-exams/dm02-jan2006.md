## Opgave 1 (20%)

### 1a — Double hashing insert (7%)

**Problem.** Open-addressing hash table, 11 slots. Current state: slot 3 = 14, slot 7 = 3, slot 9 = 41. Probe sequence
$h(k,i) = (h_1(k) + i\,h_2(k)) \bmod 11$ with $h_1(k)=k \bmod 11$, $h_2(k)=1+(k \bmod 10)$. Insert key 18. Which slot does it land in?

**Type.** Trace double hashing (open addressing) probe sequence until a free slot.

**Method (any instance).**
1. Compute $h_1(k)$ and $h_2(k)$ for your key.
2. For $i=0,1,2,\dots$ compute $h(k,i)=(h_1+i\,h_2)\bmod m$.
3. First slot in that sequence that is empty is the answer. Stop there.

**Worked solution.** $k=18$: $h_1=18\bmod 11=7$, $h_2=1+(18\bmod 10)=9$.
- $i=0$: $(7+0)\bmod 11 = 7$ → occupied (3).
- $i=1$: $(7+9)\bmod 11 = 16\bmod 11 = 5$ → empty.

**Answer: slot 5.**

### 1b — Binary heap insert (6%)

**Problem.** Min-heap (smallest at root). As array by levels: `[1, 3, 5, 4, 10, 13, 7, 6, 17]`. Insert priority 2. Draw the heap afterward.

**Type.** Heap insert = append at end, then sift-up (bubble up) restoring min-heap order.

**Method (any instance).**
1. Append the new key at the next free leaf (end of array).
2. While it is smaller than its parent, swap with the parent.
3. Stop at the root or when parent is no longer larger.

**Worked solution.** Array has 9 elements; new element 2 goes to index 9 (0-based), i.e. the 10th slot. Parent index $=\lfloor(10-1)/2\rfloor = 4$ (1-based view: child 10, parent 5). The current node at that position is 10 (right child of 3, the node 10 in the tree).
- 2 vs parent 10 → swap. New parent is 3 (the left child of root).
- 2 vs 3 → swap. New parent is root 1.
- 2 vs 1 → stop (1 < 2).

Resulting heap array: `[1, 2, 5, 4, 3, 13, 7, 6, 17, 10]`. As a tree:

```
              1
        2            5
    4      3      13     7
  6  17  10
```

So 2 sits as the left child of the root; 3 moves down to be the left child of 2; 10 becomes the left child of 3.

### 1c — Solve recurrence (7%)

**Problem.** $T(n) = 3T(n/3) + n^2$. Give the asymptotic solution.

**Type.** Solve divide-and-conquer recurrence with the Master theorem.

**Method.** For $T(n)=aT(n/b)+f(n)$: compare $f(n)$ with $n^{\log_b a}$.
- Case 1: $f = O(n^{\log_b a - \varepsilon})$ → $\Theta(n^{\log_b a})$.
- Case 2: $f = \Theta(n^{\log_b a})$ → $\Theta(n^{\log_b a}\log n)$.
- Case 3: $f = \Omega(n^{\log_b a + \varepsilon})$ and regularity $af(n/b)\le cf(n)$ → $\Theta(f(n))$.

**Worked solution.** $a=3,\ b=3 \Rightarrow \log_b a = \log_3 3 = 1$, so $n^{\log_b a}=n$. $f(n)=n^2 = \Omega(n^{1+1})$, case 3. Regularity: $3\,(n/3)^2 = n^2/3 \le c\,n^2$ with $c=1/3<1$. Holds.

**Answer: $T(n)=\Theta(n^2)$.**

## Opgave 2 (25%) — Inversions

An inversion is a pair $(x,y)$ with $x>y$ and $x$ before $y$ in the sequence.

### 2a — Max number of inversions (6%)

**Problem.** Largest number of inversions in a sequence of $n$ numbers? What does such a sequence look like?

**Type.** Counting / extremal argument.

**Worked solution.** Every unordered pair of positions $\{i,j\}$ is at most one inversion, and there are $\binom{n}{2}$ pairs. The maximum is reached when *every* pair is an inversion, i.e. the sequence is strictly decreasing (sorted in reverse). 

**Answer: $\dfrac{n(n-1)}{2}$ inversions, for a sequence sorted in descending order.**

### 2b — Count inversions with modified mergesort (7%)

**Problem.** Modify mergesort so it returns the number of inversions, keeping $O(n\log n)$. Only count them, don't list them. Assume distinct values.

**Type.** Augment divide-and-conquer (count during merge).

**Method / idea.** Inversions split into three groups: those fully in the left half, fully in the right half, and *crossing* (left element larger than a right element). Sort-and-count recursively on each half, then count the crossing inversions while merging.

Key observation in `Merge`: when you take an element from the **right** half before the left half is empty, that right element is smaller than *all remaining* left-half elements. Since the left half is sorted, those remaining left elements are all greater and all sit before it in the original order → each is an inversion. Add the count of remaining left elements.

```
SortAndCount(A):
    if |A| <= 1: return (A, 0)
    split A into L, R
    (L, cL) = SortAndCount(L)
    (R, cR) = SortAndCount(R)
    (A, cM) = MergeAndCount(L, R)
    return (A, cL + cR + cM)

MergeAndCount(L, R):
    i = j = 0; count = 0; out = []
    while i < |L| and j < |R|:
        if L[i] <= R[j]:
            out.append(L[i]); i++
        else:                       # R[j] < L[i]
            out.append(R[j]); j++
            count += |L| - i        # all remaining L > R[j]
    append the rest of L or R
    return (out, count)
```

**Running time.** Same recurrence as mergesort, $T(n)=2T(n/2)+O(n)=O(n\log n)$. The counting adds only $O(1)$ per merge step.

### 2c — Best sort given inversion count (12%)

**Problem.** Pick the sort with smallest asymptotic worst-case time in three cases.

**Method.** Match the structure of the input to a sort that exploits it. "Almost sorted" (few inversions) → insertion sort runs in $O(n + I)$ where $I$ is the inversion count. Small integer range → counting/radix sort. General → mergesort/heapsort.

1. **$O(n)$ inversions.** Insertion sort runs in $\Theta(n + I)$ time, where $I$ = number of inversions (each inner-loop shift removes exactly one inversion). With $I=O(n)$ that is $O(n+n)=O(n)$. **Use insertion sort, $O(n)$.**

2. **$O(n\log n)$ inversions, integers in $0..n^2-1$.** Inversion bound would give insertion sort $O(n\log n)$ — same as comparison sorts, no win, and we can do better with the range info. Keys are integers in $0..n^2-1$. **Radix sort** with base $n$: two digits, each digit a counting sort over $n$ values. Time $\Theta(d\,(n+b)) = \Theta(2(n+n)) = \Theta(n)$. **Use radix sort, $O(n)$.**

3. **$O(n^2)$ inversions.** This is the worst case (no structure to exploit — $n^2$ is the max). Insertion sort would be $\Theta(n^2)$. **Use a comparison sort with worst-case $O(n\log n)$: mergesort or heapsort. $O(n\log n)$.**

## Opgave 3 (30%) — Weighted graphs

Graph edges (Figure 1a): A-B=2, B-C=7, A-D=6, B-D=6, B-E=5, C-F=5, D-E=5, E-F=1, D-G=1, D-H=4, E-H=3, F-I=4, H-I=2.

First 5 Kruskal edges (drawn bold, matching forest 1b): E-F(1), D-G(1), A-B(2), H-I(2), E-H(3). The four sets after these are $\{E,F,H,I\}$, $\{D,G\}$, $\{A,B\}$, $\{C\}$.

### 3a — Kruskal next edge + union-find forest (8%)

**Problem.** After the first 5 edges, which edge does Kruskal add next? Show the disjoint-set forest after adding this 6th edge, using union by rank + path compression.

**Type.** Trace Kruskal one step + union-find with union-by-rank and path compression.

**Method.**
1. Sort remaining edges by weight, scan in order.
2. Next chosen edge = the lightest edge whose endpoints are in different sets (find-roots differ).
3. To union: compare ranks, attach the lower-rank root under the higher-rank root; if equal, pick one and increment its rank by 1. Path compression flattens nodes on a `find` path to the root.

**Worked solution.** Remaining edges by weight after the first 5: D-H(4), F-I(4), B-E(5), C-F(5), D-E(5), A-D(6), B-D(6), B-C(7). Scan:
- D-H(4): D is in $\{D,G\}$, H is in $\{E,F,H,I\}$ — different sets. **Chosen.**

**Next edge: D-H (weight 4).**

Forest state, replayed with union-by-rank + path compression in the exact Kruskal order (parent / rank):

Starting forest = Figure 1(b): E (rank 2) with children F and I, and H below I (H → I → E); D (rank 1) with child G; A (rank 1) with child B; C alone (rank 0).

Union D-H: Kruskal calls `Union(Find(D), Find(H))`. `find(D)=D` (rank 1). `find(H)` walks H → I → E, so `find(H)=E` (rank 2) — and **path compression flattens both H and I to point straight at E**. Then union by rank attaches the lower-rank root D under E (1 < 2). Ranks unchanged.

Resulting forest after the 6th edge (checked: Python):

```
Set {E,F,I,H,D,G}, root E (rank 2):
          E
        / | \ \
       F  I  H  D
                 \
                  G

Set {A,B}, root A (rank 1):   B -> A
Singleton {C}, root C (rank 0)
```

The key point most people miss: the `find(H)` call path-compresses, so the old H → I link disappears. After it, H and I both point directly at E. D (carrying its child G) attaches under E; E stays rank 2.

### 3c — Does an MST contain a given edge set? (8%)

**Problem.** Graph $G$, edges $\{e_1,\dots,e_k\}$. Decide whether some MST of $G$ contains all $k$ edges. State the running time.

**Type.** MST property / algorithm design.

**Method and answer.**
1. First the $k$ edges must form **no cycle**. If $\{e_1,\dots,e_k\}$ contains a cycle, no spanning tree (let alone MST) contains all of them → answer No. Check with union-find in $O(k\,\alpha)$.
2. Force the edges in, then complete greedily. Contract / union the endpoints of all $e_i$ (treat the $k$ edges as already chosen, total forced weight $W_0$). Then run Kruskal on the *remaining* edges to finish a spanning tree, getting total weight $W = W_0 + (\text{Kruskal completion})$.
3. Compare $W$ with the weight $W^*$ of a real MST of $G$ (run Kruskal/Prim normally). If $W = W^*$, then yes, an MST containing all $k$ edges exists; otherwise no.

Reason it works: forcing the edges and greedily completing gives the lightest spanning tree among those containing $\{e_1,\dots,e_k\}$ (exchange argument). That tree is an MST of $G$ iff its weight equals the global minimum $W^*$.

**Running time.** Two Kruskal runs dominate: sorting edges $O(E\log E)$ plus union-find near-linear, so **$O(E\log E)=O(E\log V)$**.

## Opgave 4 (25%) — Dynamic programming, string interleaving

$z$ is a "flet" (interleaving / merge) of $x$ and $y$ if $z$ can be formed by interleaving the characters of $x$ and $y$ keeping each one's order. The given recursive `StrengFlet` decides this. `Flet(x,y,z,i,j)` is true iff the prefix $z[1..i+j]$ is an interleaving of $x[1..i]$ and $y[1..j]$.

Here $x=$ `akt` $(n=3)$, $y=$ `beate` $(m=5)$, $z=$ `abekatte` (length 8).

### 4a — Fill the truth table (8%)

**Problem.** Fill in `Flet(akt, beate, abekatte, i, j)` for $i=0..3$, $j=0..5$.

**Type.** Fill DP table for interleaving (`isInterleave`).

**Method / recurrence.** $T[0][0]=\text{true}$. For other $(i,j)$:
$T[i][j] = (i>0 \wedge z[i+j]=x[i] \wedge T[i-1][j]) \ \vee\ (j>0 \wedge z[i+j]=y[j] \wedge T[i][j-1])$.

**Worked solution** (S = true/Sand, F = false/Falsk):

| i\\j | 0 | 1 | 2 | 3 | 4 | 5 |
|------|---|---|---|---|---|---|
| **0** | S | F | F | F | F | F |
| **1** | S | S | S | F | F | F |
| **2** | F | F | S | S | S | F |
| **3** | F | F | F | S | S | S |

$T[3][5]=$ S, so `abekatte` is an interleaving of `akt` and `beate` — matches the example.

### 4b — How many calls to Flet(akt,beate,abekatte,2,3)? (8%)

**Problem.** When running `StrengFlet(akt, beate, abekatte)` (the naive recursion as written, no memoization), how many times is `Flet(...,2,3)` called?

**Type.** Count recursive calls of an un-memoized recursion (respect short-circuit $\vee$).

**Method.** Follow the exact pseudocode: `Fx` checks `z[i+j]=x[i]` and only recurses if that char matches; `Fy` likewise. `∨` short-circuits — if `Fx` returns true the `Fy` branch is skipped.

**Worked solution.** Trace from `Flet(3,5)`:
- `Flet(3,5)`: `Fx(3,5)`: $z[8]=$`e`, $x[3]=$`t` → false (no recursion). `Fy(3,5)`: $z[8]=$`e`, $y[5]=$`e` → recurse `Flet(3,4)`.
- `Flet(3,4)`: `Fx(3,4)`: $z[7]=$`t`, $x[3]=$`t` → recurse `Flet(2,4)`.
- `Flet(2,4)`: `Fx(2,4)`: $z[6]=$`t`, $x[2]=$`k` → false. `Fy(2,4)`: $z[6]=$`t`, $y[4]=$`t` → recurse **`Flet(2,3)`** (call #1).
- `Flet(2,3)` then descends $\to (2,2)\to(1,2)\to(1,1)\to(1,0)\to(0,0)$, all returning true, so everything short-circuits back up. No other branch ever reaches $(2,3)$.

**Answer: exactly 1 call.** (The unique successful interleaving path passes through $(2,3)$ once; short-circuit `∨` prunes every alternative before any of them re-reaches $(2,3)$.)

### 4c — Iterative DP algorithm + running time (9%)

**Problem.** Write, in pseudocode, an iterative dynamic-programming algorithm deciding whether $z$ is an interleaving of $x$ and $y$. Give the running time.

**Type.** Bottom-up DP from the recurrence in 4a.

**Method.** Build the boolean table $T[0..n][0..m]$ bottom-up using the recurrence, return $T[n][m]$.

```
Interleave(x, y, z):
    n = |x|; m = |y|
    if n + m != |z|: return False
    let T[0..n][0..m] be a boolean table
    T[0][0] = True
    for i = 0 to n:
        for j = 0 to m:
            if i == 0 and j == 0: continue
            v = False
            if i > 0 and z[i+j] == x[i] and T[i-1][j]: v = True
            if j > 0 and z[i+j] == y[j] and T[i][j-1]: v = True
            T[i][j] = v
    return T[n][m]
```

**Running time.** $(n+1)(m+1)$ cells, $O(1)$ work each → **$\Theta(nm)$** time, $\Theta(nm)$ space (reducible to $\Theta(\min(n,m))$ with a rolling row). This removes the exponential blow-up of the naive recursion.
